# Incident: v2.1.1 release — cascading deployment failures after directory rename

**Date**: 2026-05-27
**Severity**: High
**Status**: Resolved

## What happened

Deploying v2.1.1 to Heroku resulted in a complete production outage lasting approximately 45 minutes. The app either failed to build or crashed on startup through a chain of four separate issues, all stemming from the rename of the `tmbr/` directory to `tmbr-web/` in PR #160. After each fix the app either continued to fail or exposed the next issue underneath.

From a user's perspective: tmbr.me returned 503 for the entire duration.

## Root cause

Four independent places still referenced the old `tmbr` directory name. None were caught before the release because the last successful Heroku deploy (v61) predated the rename.

### Issue 1: `PACKAGE_DIR` Heroku config var
The Vapor buildpack uses `PACKAGE_DIR` to locate the Swift package inside the repo. It was set to `tmbr` on Heroku; after the rename to `tmbr-web` the build failed immediately with `cd: /tmp/.../tmbr: No such file or directory`.

**Fix**: `heroku config:set PACKAGE_DIR=tmbr-web`

### Issue 2: `import CoreGraphics` in `TmbrCore`
When `ImageResponse` was moved to `tmbr-core` (PR #160), `CGSize` and `import CoreGraphics` came with it. `CoreGraphics` doesn't exist on Linux. It accidentally compiled before because `ImageResize` made `CGSize` available across the whole `App` module in `tmbr-web`, but `tmbr-core` is a separate module with no such dependency.

**Fix**: Replaced `CGSize` with a plain cross-platform `ImageSize` struct (`width: Double, height: Double`) in `tmbr-core`. JSON encoding is identical since field names match.

### Issue 3: Missing `SIWA_NATIVE_APP_ID` env var
PR #163 added `nativeAppID = Environment.get("SIWA_NATIVE_APP_ID")!` to `Environment+SignIn`. The force-unwrap caused a `fatalError` on startup because `SIWA_NATIVE_APP_ID` was never added to Heroku's config. The app crashed with SIGILL immediately after the new binary launched.

**Fix**: `heroku config:set SIWA_NATIVE_APP_ID=me.tmbr`

### Issue 4: Hardcoded `workingDirectory: "tmbr"` in renderer config
`Configuration+Renderer.swift` set Vapor's working directory to `"tmbr"` in production. Leaf resolves template paths relative to this directory. With the directory renamed to `tmbr-web`, all template lookups failed — the app started but every request returned a JSON error (`No template found for Posts/posts`).

**Fix**: Changed `workingDirectory: "tmbr"` to `workingDirectory: "tmbr-web"`.

## How it was fixed

Four separate commits and config changes, applied in discovery order:
1. Heroku config var `PACKAGE_DIR` updated to `tmbr-web`
2. `TmbrCore/Responses/ImageResponse.swift` — `CGSize` replaced with `ImageSize`
3. Heroku config var `SIWA_NATIVE_APP_ID` added (`me.tmbr`)
4. `Configuration+Renderer.swift` — working directory updated to `tmbr-web`

## Prevention

- [ ] **Pre-release checklist**: Before any Heroku deploy, grep the codebase for the old directory name and verify all Heroku config vars against `Environment.get(...).force-unwrapped` calls. Add this to the release process in `quality-assurance.md`.
- [ ] **New env var policy**: Any `Environment.get(...)!` added in a PR must be accompanied by a note in the PR description listing the new var and its expected value. The deployer is responsible for setting it before the release.
- [ ] **Cross-platform compile check**: `tmbr-core` is a shared Linux/Apple package. Add a CI step (or local `swift build` on Linux via Docker) to catch Apple-only framework imports (`CoreGraphics`, `UIKit`, `AppKit`) before they reach Heroku.
- [ ] **Staging environment**: A staging Heroku app that tracks `main` would have caught issues 1, 3, and 4 automatically on merge rather than during the manual release.

## Tests added

None — all four issues are configuration/environment problems that tests wouldn't catch directly. The process changes above are the primary mitigation.
